  <behavior>
    - buildInsightId("R1", "2026-03-15") returns the SHA256 hex digest of the string "R1-2026-03-15"
    - buildInsightId produces the same output when called twice with identical inputs (deterministic)
    - persistInsights([]) returns { upserted: 0, errors: [] } immediately
    - persistInsights with one valid candidate upserts one row and returns { upserted: 1, errors: [] }
    - On upsert conflict (same id), generated_at / severity / evidence are updated BUT dismissed_at is NOT overwritten when it is already set (dismissed rows stay dismissed)
    - A Supabase error for one candidate is caught, appended to errors[], and does not abort remaining candidates
  </behavior>
  <action>
    Create `lib/persist-insights.ts`:

    1. Import `createHash` from Node's built-in `"crypto"` module (no extra package needed).
    2. Import `createAdminClient` from `"@/lib/supabase/admin"`.
    3. Import `InsightCandidate`, `PersistResult` from `"@/lib/intelligence-types"`.

    **buildInsightId function:**
    ```typescript
    export function buildInsightId(ruleId: string, calendarDate: string): string {
      return createHash("sha256").update(`${ruleId}-${calendarDate}`).digest("hex");
    }
    ```
    `calendarDate` is the ISO date string in YYYY-MM-DD format (e.g. `new Date().toISOString().slice(0, 10)`).

    **persistInsights function:**
    ```typescript
    export async function persistInsights(
      candidates: InsightCandidate[],
    ): Promise<PersistResult>
    ```

    Implementation steps:
    - Compute today's calendar date: `const today = new Date().toISOString().slice(0, 10);`
    - Initialize result: `{ upserted: 0, errors: [] as string[] }`
    - For each candidate in a for-loop (not Promise.all — sequential is fine and avoids rate issues):
      - Compute `id = buildInsightId(candidate.rule_id, today)`
      - Build the upsert row: `{ id, rule_id, severity, module_tags, evidence, generated_at: new Date().toISOString() }`
      - Call `supabase.from("cross_module_insights").upsert(row, { onConflict: "id", ignoreDuplicates: false })`
      - The upsert updates generated_at, severity, and evidence on conflict.
      - IMPORTANT: dismissed_at must NOT be included in the upsert payload. Supabase's upsert only updates columns present in the payload — omitting dismissed_at preserves the existing value, which is the correct behavior (dismissed insights stay dismissed).
      - Wrap each upsert in try/catch: on error, push `"${candidate.rule_id}: ${error.message}"` to result.errors; on success, increment result.upserted.
    - Return result.

    Do NOT use `supabase.from(...).insert()` — upsert is required for idempotency.
  </action>
